# Gotchas：项目中发现的所有陷阱

> 不是设计文档。是踩过的坑。

---

## 一、蒸馏层

### Gotcha 1：AI 会自然收敛到"温和正确"——这不是巴菲特

**现象**：v1-v2 的 Skill 回答严谨、周全、边界清晰——但太平淡。真正的巴菲特在年报里引经据典，在 CNBC 上毒舌犀利。

**根因**：SKILL.md 写的是"谨慎、圆融、留有余地"——AI 把"谨慎"理解成了"不犯错"，把"留有余地"理解成了"不说任何尖锐的话"。

**修复**：v3.0 加入三个强制厚度维度（纵向沿革+横向引证+反面教训）+ `/crazy` 模式。不是让 AI "更激进"，是让它"更有信息密度"。

### Gotcha 2：'保留标签，替换内容'——巴菲特的认知策略也是蒸馏陷阱

**现象**：巴菲特自称"85% 格雷厄姆 + 15% 费雪"。但批次 18 逐条对比原文后发现——安全边际的名称没变，操作定义全换了：格雷厄姆的是数学比率（E/P - 债券利率 = 5%），巴菲特的是定性判断（"明显低于"）。

**根因**：思想家会**高估**导师的贡献以保持敬意，同时**低调**地替换了导师的概念内容。蒸馏时如果只看思想家的自述，会继承这个偏差。

**修复**：蒸馏流程必须同时读思想家的自述（年报）和导师的原著（格雷厄姆原文）——交叉对比才能发现"标签没变内容换了"。

### Gotcha 3：概念卡片格式 ≠ 概念卡片质量

**现象**：validate-evidence.py 给了满分，但卡片中的引用可能没有精确匹配原文——只是"大意对了"。

**根因**：校验脚本检查形式（有没有引用？有没有章节号？），不检查内容（引用的内容和原文一致吗？）。

**修复**：加了第四层 `/factcheck` 引用溯源脚本。但它的精度受限于素材是否在本地存在（extracted epub 在 .gitignore 里）。

### Gotcha 4：即席表达和书面表达是两种语言

**现象**：批次 15 发现——年报说"内在价值是未来现金流的折现值"，CNBC 上说"你应该能拿一张黄纸，写清楚为什么花 420 亿买通用汽车"。同样的逻辑，完全不同的语言。

**根因**：SKILL.md v1-v2 的素材全部来自年报（书面表达）。`/crazy` 模式需要的是专访语言。

**修复**：`buffett-voice-dna.md` 新增 §八 "即席表达 vs 书面表达" 双模态。`/crazy` 用即席语料。

## 二、工程层

### Gotcha 5：Git 仓库初始化晚了一周——版本节点丢失

**现象**：v1.0→v4.0 的 5 次迭代在本地完成，Git 是项目后期才初始化的。GitHub 上只有 3 个初始 commit → 后来才补 commit。

**根因**：项目启动时没有 Git。这是小项目的通病。

**修复**：README 加了"Git 历史说明"诚实声明。无法修复历史，只能保证从 v4.0 起每次改动一个 commit。

### Gotcha 6：校验脚本的检测词库需要跟着内容一起进化

**现象**：validate-framework.py 对 `2008-crisis-war-game.md` 报"素材溯源: 1/2"——但文件里全是具体数据和历史案例，只是没有用脚本预期的关键词（"Cunningham 第 X 章"）。

**根因**：脚本用正则匹配检测词。内容写得越丰富、越不拘泥于模板用语，脚本的误判率越高。

**修复**：至今未完全修复。这是一个结构性矛盾——脚本追求形式合规，好内容追求表达自由。目前的妥协是：脚本检测形式，人工判断内容。

### Gotcha 7：MCP 配置不重启不生效

**现象**：加完 yfinance MCP 配置后，Skill 仍然说"没有实时数据"。

**根因**：Claude Code 只在启动时加载 MCP 配置。改了配置文件不重启 = 没改。

**修复**：SKILL.md §〇.五 的无工具钩子提示里包含了"重启 Claude Code"——但很多用户可能跳过。这是 MCP 生态的体验问题，不是我们项目独有的。

## 三、交互层

### Gotcha 8：用户不知道 Skill 能干什么

**现象**：v4.0 之前，用户反馈"不知道该问什么""每次问你回答都差不多"——Skill 的框架强大但发现性差。

**根因**：Skill 是一个"你可以问任何投资问题"的开放式界面。开放式界面 = 用户不知道边界 = 随机试探 = 体验差。

**修复**：v4.0 加入 7 大应用场景 × 结构化输出模板。从"可以问任何问题"变成"这 7 类问题我能明确回答"。

### Gotcha 9：回答太厚，用户可能读不完

**现象**：v3.0 的纵深升级后，一个回答 400+ 字是常态。在聊天界面里，这是一堵墙。

**根因**：年报读者有耐心读长文；聊天用户没有。

**修复**：未完全解决。目前 `/crazy` 模式用"金句先出、解释后跟"的结构缓解。标准模式仍然偏厚——这是内容深度和阅读体验之间的取舍。

## 四、跨模型兼容层（预判，尚未实测）

### Gotcha 10（预判）：Sonnet 可能比 Opus 更"圆滑"

**假设**：Sonnet 在理解"确定性分层"时可能比 Opus 更保守——把 L4（极高）降级为 L3（中等），导致回答缺乏确信感。

**验证计划**：T3 跨模型测试——用同样的用例在 Sonnet 和 Opus 上各跑一次，对比回答的确定性分布。

### Gotcha 11（预判）：GPT 可能丢失第一人称

**假设**：GPT-4o 在遇到复杂推理时，可能从"我们"滑回"巴菲特认为"——丢失第一人称人设。

**验证计划**：T3 测试中包含人称一致性检查。

### Gotcha 12（预判）：中文模型可能回退到"安全边际=买便宜货"

**假设**：批次 19 识别了中文世界的理解偏差（"安全边际→买便宜货"）。如果中文模型（如 DeepSeek/GLM）运行这个 Skill，可能因为训练数据中的偏差而回退到格雷厄姆的烟蒂版定义。

**验证计划**：T3 测试中包含"安全边际"概念的跨模型一致性检查。

## 总结：如果重做这个项目，会改什么

1. **Day 1 就初始化 Git**——版本节点不会丢失
2. **先定义 gotchas 再写 SKILL.md**——而不是反过来
3. **校验脚本和内容同步迭代**——而不是内容先走了，脚本还在用旧检测词
4. **第一天就设计场景模板**——用户不需要"万能的 Skill"，需要"明确的问题 → 明确的输出格式"
5. **蒸馏流程标配第三方传记对照**——思想家的自述总会美化自己的思想来源
