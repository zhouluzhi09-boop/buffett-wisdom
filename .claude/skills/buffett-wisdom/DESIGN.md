> 本文档回答"为什么这样做"而非"做了什么"。
> "做了什么"见项目 README 和 SKILL.md 本身。

---

## 1. 为什么不是 RAG？为什么不是向量数据库？

### RAG（检索增强生成）的经典范式

```
用户问题 → Embedding → 向量检索 Top-K 文档片段 → LLM 生成回答
```

### 为什么这个项目不需要 RAG

巴菲特的价值不在于**他说过什么具体的句子**（那是原话检索），而在于**他怎么思考**（那是认知框架）。

RAG 能告诉你"巴菲特在 1977 年年报第 12 页写过：安全边际是……"。但它不能告诉你"面对一个巴菲特从未见过的 2026 年的科技公司，他会怎么分析"。前者是句子匹配，后者是认知模仿。

| 能力 | RAG 方案 | 我们方案 |
|------|:--:|:--:|
| 找到巴菲特的原话 | ✅ 很强 | ⚠️ 依赖预训练记忆 |
| 模仿巴菲特的推理路径 | ❌ 做不到 | ✅ 核心能力 |
| 承认框架的边界 | ❌ 做不到 | ✅ L3 演化层 |
| 被社区持续修正 | ❌ 做不到 | ✅ 模板+命令+校验 |
| 跨场景一致 | ❌ 碎片化回答 | ✅ 同一框架约束 |

### 为什么不用向量数据库

向量数据库解决的是**海量文本的相似性检索**。我们的 10 张概念卡片 + 5 份认知操作，总共约 150KB 的纯文本——远不到需要向量检索的量级。把 150KB 做成 embedding + 向量库是杀鸡用牛刀。

更根本的问题：即使向量检索完美召回了所有相关段落，它仍然不知道**段落之间的逻辑关系**。知道"护城河"和"安全边际"的定义 ≠ 知道"先判断护城河再判断安全边际"这个决策顺序。后者是 L2 认知层专门做的事。

### 未来可能用 RAG 的地方

当社区 L3 贡献足够多时——比如积累了几百条新情景分析、修正提案、辩论记录——那时向量检索可能有用。但目前的内容量，一个纯文本的层级结构足够。

## 2. 为什么三层架构（L1→L2→L3）而不是平面知识库？

### 平面知识库的问题

把所有内容平铺在一个"知识库"里——比如一个巨大的 FAQ 文档或一个维基——会导致三个问题：

1. **层级丢失**："安全边际 = 内在价值 - 价格"和"巴菲特在 1972 年禧诗糖果收购中学会了护城河"是两个不同抽象层级的知识，平铺会抹平这种差异
2. **不可推演**：平面知识只能回答"已知问题"。遇到新场景——比如"比特币符合巴菲特的投资标准吗"——平面知识库只能检索到"巴菲特说过比特币是老鼠药的平方"这个结论，无法推理出"为什么不产生现金流的东西不在框架内"
3. **不可演化**：平面知识库改一条规则可能破坏一百条隐含依赖。三层的每一层有自己的校验脚本，改 L1 不会意外破坏 L2

### 三层分别解决什么问题

```
L1 蒸馏：     巴菲特说过什么？
              → 把原文转化为结构化概念卡片
              → 校验：每条规则可追溯到原文出处

L2 认知：     巴菲特遇到没见过的事会怎么想？
              → 从卡片中提取决策流程、优先级排序、排除模式
              → 校验：流程可用于推演历史场景（2008）且结果一致

L3 演化：     这个框架有什么盲区？谁可以修正它？
              → 标注边界、不适用场景、待社区回答的问题
              → 校验：红队对抗测试
```

### 为什么不是两层或四层

- **两层（L1+L2）不够**：没有盲区声明，框架变成"万能的"——这恰恰是大多数投资类 Prompt 的致命问题。承认边界比假装全能更可信。
- **四层（L1+L2+L3+L4）多余**：L3 已经是"开放演化"。再加一层（如 L4=社区治理）在当前规模下是过度设计。等社区贡献量真正需要治理规则时再分离。

### 与 Karpathy LLM Wiki 模式的关系

三层架构是对 Karpathy `raw/ → wiki/ → CLAUDE.md` 模式在 Skill 领域的特化：
- `raw/` 对应我们的原始素材层
- `wiki/` 被拆分为 L1（概念卡片）+ L2（认知操作）
- `CLAUDE.md` 被升级为 SKILL.md（不仅是模式文件，而是可执行的 AI 指令）

## 3. 为什么 7 个场景而不是更多/更少？

### 设计原则：覆盖主航道，舍弃边缘

场景模板的设计遵循两个约束：
1. **每个场景必须有独立的输出模板**——不是"遇到什么都可以问"，而是"遇到这 7 类问题有明确格式"
2. **一个场景如果和另一个场景的输出格式无法清晰区分，就应该合并**

### 7 个场景的划分逻辑

| 场景 | 划分依据 | 为什么不能合并 |
|------|---------|-------------|
| 公司初判 | 用户给了一个具体标的 | 和"估值判断"不同——先得判断看不看得懂，才能判断贵不贵 |
| 市场恐慌/狂热 | 用户带着情绪提问 | 需要先安抚情绪再给框架——输出顺序和其他场景不同 |
| 商业决策 | 用户在做一个操作选择 | 有明确的五级优先级排序——不是开放式分析 |
| 管理层判断 | 用户对人有疑问 | 用嫁女儿测试+红旗信号——这种人格化判断和其他场景的分析方式完全不同 |
| 估值判断 | 用户只关心价格 | 这是最常被问的问题——独立出来让用户知道"估值不精确" |
| 投资哲学/人生 | 用户不持有一个具体标的 | 这是"怎么活"的问题，不是"怎么投"的问题 |
| 框架极限 | 用户问超出框架的事 | 这个场景的存在本身就是边界声明——"我诚实地说这个我不懂" |

### 为什么不加第 8 个（如"行业分析"）

"行业分析"可以被场景 1（公司初判）吸收——护城河那一步自然包含了行业分析。单独列出来反而让用户困惑："问行业分析和问公司初判有什么区别？"

### 7 的"恰到好处"

- 3 个太少：覆盖不全主要使用模式
- 15 个太多：用户记不住，本质上是把"回答问题"变成了"查手册"
- 7 个适合：用户可以在心里列出来——"我是类型 2 的恐慌问题"

## 4. 五级确定性体系的边界是怎么定的？

### 为什么是五级

```
L5（绝对）: never, always, must     ← 核心原则和道德底线
L4（极高）: I believe, almost        ← 基于长期经验但留一丝余地
L3（默认）: we think, in our view    ← 有依据但不绝对
L2（极少）: perhaps, might           ← 推测性陈述（巴菲特很少用）
L1（高频）: I don't know             ← 认知诚实的核心
```

五级不是随意划分的。它来自于对巴菲特原文的统计分析：

- **L5** 仅出现于三类语境：投资基本原则（"永远不借钱"）、道德声明（"名誉胜过金钱"）、数学确定性（"零乘任何数等于零"）
- **L4** 用于经过长期验证的经验法则——巴菲特说"我相信"时，通常是他亲自验证过几十年的事
- **L3** 是默认语气——不绝对、不犹豫、坦率。这是巴菲特的"中性"状态
- **L2** 极少出现——巴菲特几乎不做"可能"式推测。出现频率低到几乎可以忽略
- **L1** 高频出现——"我不知道"是巴菲特最被低估的能力

### 为什么不是更简单的三级（确定/不确定/不知道）

三级太粗糙。L4（极高）和 L3（默认）之间的差异非常重要——"我相信好企业比便宜价格重要"和"以我们的观点看当前估值不便宜"是两种完全不同的确信程度。前者是八十年的经验，后者是当前的判断。把它们混为一谈会让 Skill 的输出失去精度。

### 为什么不对标学术的置信度模型

学术模型的问题是：它要求给出一个概率（"我认为有 80% 的概率…"）。巴菲特从不这样说——他整个框架的精髓就是**拒绝假装能量化不确定性**。"模糊的正确胜过精确的错误"恰恰反对给不确定性赋概率值。五级确定性是一种**定性的、可沟通的**分级，而非量化的概率模型。

## 5. `/crazy` 模式的产品决策——为什么需要两种人格？

### 问题起源

v2 测试后发现，Skill 的回答"过于四平八稳"——严谨、正确、但缺乏锐度。真实的巴菲特在 CNBC 专访和股东年会即席问答中，比年报中犀利得多。

### 为什么不直接改默认模式

如果直接把默认模式改成犀利风格，会失去年报巴菲特的"厚重感"——纵向历史深度、横向引经据典、谨慎的确定性分层。年报巴菲特和 CNBC 巴菲特是两种不同的沟通场景，各有价值。前者适合"教你怎么想"，后者适合"直接告诉你怎么看"。

### 为什么不分开成两个独立 Skill

两种人格共享同一套核心原则、决策流程和边界约束。把它们拆成两个 Skill 会导致：
- 概念卡片要维护两份
- 校验脚本要维护两份
- 任何框架修正要改两处

一个 Skill 带模式切换是最小维护成本的设计。

### 触发方式的设计选择

内部关键字触发（`/crazy`）而不是自动场景判断（"检测到情绪激动的问题就切换"），原因是：
- 用户应该明确知道自己在和"哪种巴菲特"对话
- 自动切换可能导致用户困惑——"刚才还是温和的，怎么突然毒舌了？"
- 显式命令让切换本身成为用户的主动选择

## 6. v1→v4 迭代了什么？每次改了什么？为什么？

### v1.0（批次 10 产出）

**状态**：功能完整但表达平淡。9 章节，从角色定义到元规则。
**问题**：回答像 FAQ——正确但缺乏厚度。

### v2.0（批次 12 产出）

**改动**：
- 扩展触发词 10 个（覆盖"股市""选股""持仓"等口语）
- 新增边界声明 2 条（杠杆、承诺回报）

**为什么**：26 个测试用例审查发现两处缺陷——T0-04"市场恐慌"可能不触发，T1-07/T1-08 缺少显式边界声明。这是在"代码审查"阶段发现的 bug。

### v3.0

**改动**：
- 标准模式深度升级：纵向沿革+横向引证+历史锚点，三个强制维度
- 新增 `/crazy` 犀利模式

**为什么**：用户反馈"回答过于四平八稳，观点不够有锐度"。溯源后发现：不是内容缺失，是**风格约束太紧**。标准模式需要的是"厚重"而非"激昂"——纵向沿革（这个概念经历了什么演变）和横向引证（这个概念和其他领域有什么关联）是真正的解决方案。`/crazy` 是对"还需要锐度"这个需求的独立响应——不改默认模式，加开关。

### v3.1

**改动**：新增 `〇.五 金融数据工具（MCP）` — 有工具自动调，无工具温和提示

**为什么**：用户问"是否可以加入 MCP 能够让巴菲特基于最新的金融信息进行回答"。这是 Skill 从"纯知识框架"到"知识框架+实时数据"的升级。关键设计决策：不强制 MCP——没有配置时不影响使用，只是温和建议。这保证了 Skill 的环境兼容性。

### v4.0

**改动**：新增 `〇.六 应用场景与输出预期` — 7 大场景 × 结构化输出模板

**为什么**：对标毛泽东 Skill 的决策引擎设计——"四渡赤水→指挥逻辑"式的场景映射。用户反馈揭示了核心问题：Skill 有强大的框架，但用户不知道**什么时候用它、用了能得到什么**。7 个场景模板解决了"发现性"问题。从"一个可以问任何投资问题的 AI"变成"一个对 7 类具体场景有明确输出格式的决策引擎"。

### 迭代规律

```
v1: 功能正确但平淡       → 测试发现
v2: Bug 修复             → 代码审查
v3: 风格深度+锐度开关     → 用户反馈
v4: 场景化+降低使用门槛   → 对标竞品分析
```

每次迭代的驱动源不同——测试、审查、反馈、竞品——说明质量保障体系需要多层覆盖。

## 7. 已知的架构缺陷和未来计划

### 缺陷 1：校验脚本是静态正则，不能检测"语义上正确但框架上不对"

`validate-skill.py` 只能检查 SKILL.md 是否**包含了正确的指令**，不能检查 Skill 实际运行时是否**按指令执行了**。例如：脚本可以检查 SKILL.md 是否写了"不推荐股票"——但不能检查 AI 是否真的在面对"腾讯能买吗"时说了"不给出买卖建议"。

**计划**：阶段 C 引入 `judge.py`——用另一个 LLM 评分 Skill 的实际输出。但这需要 LLM-as-judge 的可靠性达到可接受水平。目前 LLM-as-judge 的误判率仍高于可接受阈值。

### 缺陷 2：L3 演化层没有触发机制

`evolution/` 下有盲区目录和 2008 实战推演，但**这些材料何时被加载，没有明确的触发规则**。如果用户问了一个恰好涉及已知盲区的问题（如零利率下的 DCF），Skill 可能不会主动查阅盲区目录来增强回答。

**计划**：在 SKILL.md §〇.五 附近增加一个"盲区感知"规则：当检测到问题涉及已知盲区时，自动查阅 `evolution/blindspots/` 下的相关条目。

### 缺陷 3：依赖单一素材源（Cunningham 编译本）

10 张 L1 卡片中的 9 张主要素材来自 Cunningham 编译本。虽然 Hagstrom 和《穷查理宝典》做了交叉验证，但大量素材（CNBC 专访逐字稿、34 份年会切片、多本传记）尚未被系统蒸馏。

**计划**：后续批次用 `/distill` 命令逐步消化其他素材，特别是：
- CNBC 专访逐字稿（8 份）→ 提取"即席表达风格"和"2008/2020 危机决策"
- 年会切片（34 份）→ 提取"跨年代的回答一致性"验证
- 传记《滚雪球》→ 提取"职业生涯关键转折"补充沿革考据

### 缺陷 4：跨模型兼容性未验证

SKILL.md 以 Claude Opus 4 为首要目标编写。在 Sonnet、GPT-4o、DeepSeek 上的行为一致性尚未测试。不同模型对"第一人称"、"确定性分层"的理解可能不同。

**计划**：批处理 12 的测试体系包含 T3 跨模型测试，但当前未执行。需要至少跑一次 Sonnet/Opus 对比测试，记录模型特例到 gotchas。

### 缺陷 5：没有 A/B 测试框架

当需要决定"是把安全边际的描述从'显著低于'改成'大幅低于'更好"时，没有客观手段判断。目前只能靠人工感觉。

**计划**：短期内不解决——A/B 测试对当前项目规模来说是过度工程。当 Skill 有 1000+ 日活用户时再考虑。
